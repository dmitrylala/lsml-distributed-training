## Вводная по заданию

Давайте явно зафиксируем модель, данные, и некоторые условия для проведения экспериментов:
- Модель: EleutherAI/pythia-160m
- Датасет: wikitext-103-v1
- Метрика качества: validation perplexity (PPL)
- `seed=42`, `seq_len=512`, `global_batch_size=131072` токенов на шаг, 1 эпоха

> [!note]
> Почему фиксируем `global_batch_size` в токенах, а не в сэмплах? При изменении числа GPU, если фиксировать число сэмплов на GPU, effective batch size меняется — модели получают разное количество данных на шаг оптимизатора, и сравнение по TPS (tokens per second) будет некорректно. Фиксация в токенах гарантирует одинаковый объем данных на шаг при любом числе GPU.

`global_batch_size` в токенах можно посчитать по формуле:
```
global_batch_size = seq_len * local_batch_size * dp_size * grad_accumulation_steps
```

`local_batch_size` — обычный `batch_size`, который указывается в скрипте для обучения. В случае использования только data parallel обучения, `dp_size = n_gpus` (наш случай).

## Что нужно для сдачи задания

Артефакты, прикладываемые для сдачи:

Python-код должен позволять воспроизвести любой пункт задания из заявленных на сдачу. Если не приложена инструкция к запуску — пункт будет считаться невыполненным. Если не продемонстрирована динамика обучения или числа на обучении не соответствуют заявленным в таблицах в отчете — пункт будет считаться невыполненным.

Какие есть опции, чтобы продемонстрировать динамику обучения:
- Писать логи в файлы, в юпитер ноутбуке парсить их, рисовать графики. Логи в таком случае также прикладываются в архив
- Логировать значения в облачный wandb. В wandb есть опция шэринга ссылок, их можно прикрепить в ноутбук. Также в этом случае ожидается, что имя пространства с логами позволит вас однозначно идентифицировать (напр. фамилия-имя-lsml)
